SecureAI Security Architecture

Security & Compliance beginner security encryption architecture data-privacy infrastructure model-providers

This article provides a technical overview of SecureAI's security architecture, covering how data is encrypted at rest and in transit, where data is stored, and how model provider interactions are secured. It is intended for IT administrators, security engineers, and compliance officers evaluating or managing SecureAI deployments.

Architecture Overview

SecureAI runs on Google Cloud Platform (GCP) Cloud Run and follows a defense-in-depth approach. The architecture consists of three layers:

  1. Edge layer — Load balancing, TLS termination, DDoS mitigation, and WAF rules.
  2. Application layer — The OpenWebUI frontend, API server, authentication services, and background workers.
  3. Data layer — Encrypted databases, object storage, and audit log storage.

Each layer enforces its own access controls and encryption boundaries. A compromise at one layer does not automatically grant access to data at another.

Encryption at Rest

All stored data is encrypted using AES-256, Google Cloud's default server-side encryption standard. The following table summarizes what is stored and how it is encrypted.

Data Type Storage Service Encryption Method Key Management
Conversations Cloud SQL (PostgreSQL) AES-256 server-side Google-managed or CMEK
Uploaded documents Cloud Storage (GCS) AES-256 server-side Google-managed or CMEK
User accounts and credentials Cloud SQL (PostgreSQL) AES-256 server-side; passwords bcrypt-hashed Google-managed or CMEK
Audit logs Cloud Logging + Cloud SQL AES-256 server-side Google-managed
Database backups Cloud SQL automated backups AES-256 server-side Same key policy as live data
Secrets and API keys Secret Manager AES-256 envelope encryption Google-managed or CMEK

Customer-Managed Encryption Keys (CMEK)

Organizations that require control over their own encryption keys can use GCP's Customer-Managed Encryption Keys through Cloud KMS:

  • How it works: Your organization creates and manages encryption keys in Cloud KMS. SecureAI's storage services use these keys instead of Google-managed keys.
  • Key rotation: CMEK keys can be rotated on a schedule you define (recommended: every 90 days). Previous key versions are retained to decrypt data encrypted under older versions.
  • Key revocation: Revoking or disabling a CMEK key renders all data encrypted under that key inaccessible. This provides a cryptographic kill switch for your data.
  • Availability: CMEK is available on enterprise service agreements. Contact your account representative to enable it.

Note: CMEK adds operational responsibility. If your organization loses access to its Cloud KMS keys, SecureAI cannot decrypt your data. Ensure your key management procedures include backup and disaster recovery.

Encryption in Transit

All network communication uses TLS 1.2 or higher. SecureAI does not support unencrypted connections at any layer.

Connection Protocol Certificate Management
Browser to SecureAI TLS 1.2+ (HTTPS) Google-managed certificates via Cloud Load Balancer
API clients to SecureAI TLS 1.2+ (HTTPS) Same as browser; API tokens sent via Authorization header
SecureAI to Cloud SQL TLS 1.2+ with server certificate validation Cloud SQL Proxy or direct SSL; certificates managed by GCP
SecureAI to Cloud Storage TLS 1.2+ (HTTPS) GCP service account authentication with short-lived tokens
SecureAI to AI model providers TLS 1.2+ (HTTPS) Provider-managed certificates; SecureAI validates certificate chains
Internal service-to-service mTLS where supported by Cloud Run Managed by GCP's internal certificate authority

TLS Configuration Details

  • Cipher suites: SecureAI accepts only cipher suites that provide forward secrecy (ECDHE key exchange). Weak ciphers (RC4, DES, 3DES, export-grade) are rejected.
  • HSTS: HTTP Strict Transport Security is enabled with a one-year max-age and includeSubDomains.
  • Certificate transparency: All public-facing certificates are logged to Certificate Transparency logs.
  • HTTP to HTTPS redirect: All HTTP requests on port 80 are redirected to HTTPS on port 443 with a 301 response.

Storage Locations

SecureAI data resides in GCP regions selected during onboarding. All data for a given organization is stored within a single region.

Region GCP Location Use Case
United States (default) us-central1 (Iowa) Default for all new organizations
European Union europe-west1 (Belgium) Organizations with EU data residency requirements
Additional regions Available on request Enterprise agreements with specific sovereignty needs

What "Single Region" Means

When your organization is assigned to a region:

  • Conversations, documents, and user data are stored in Cloud SQL and Cloud Storage instances in that region.
  • Backups remain in the same region (GCP regional backups).
  • Audit logs are stored in Cloud Logging within the same region.
  • CDN caching: Static frontend assets (JavaScript, CSS, images) may be cached at edge locations globally for performance. These assets contain no customer data.

Cross-Region Data Flows

Two types of data leave your assigned region by design:

  1. AI model provider requests: When a user sends a message, the conversation context is transmitted to the configured model provider's API endpoint. The provider's infrastructure may be in a different region. See Model Provider Data Handling for details.
  2. DNS and load balancing metadata: GCP's global load balancer processes routing metadata (IP addresses, request headers) outside the data region. This metadata does not include conversation content or document data.

No other customer data leaves the assigned region.

Model Provider Data Handling

SecureAI acts as a gateway between your users and AI model providers. Understanding how data flows through model providers is critical for security and compliance assessments.

Data Flow

User → SecureAI API → [Prompt Construction] → Model Provider API → [Response] → SecureAI API → User
  1. User sends a message via the browser or API.
  2. SecureAI constructs the prompt, which includes the user's message and relevant conversation history (within the model's context window).
  3. SecureAI sends the prompt to the configured model provider's API over a TLS-encrypted connection.
  4. The model provider processes the prompt and returns a response.
  5. SecureAI receives the response, stores it in the conversation history, and returns it to the user.

What Is Sent to Model Providers

Data Sent Details
Current message The user's latest input
Conversation context Prior messages in the current conversation, up to the model's context window limit
System prompt Organization-configured system instructions (if set by an administrator)
Uploaded document content Text extracted from documents referenced in the conversation (sent as context, not as file uploads)

What Is NOT Sent to Model Providers

Data Not Sent Explanation
User identity User names, emails, and account details are stripped before the request leaves SecureAI
Other users' conversations Each request contains only the current user's conversation context
Organization metadata Organization names, billing information, and configuration settings are not included
Other organizations' data Tenant isolation is enforced before prompt construction

Model Provider Agreements

SecureAI maintains data processing agreements with each supported model provider. These agreements establish:

  • No training on customer data: Providers are contractually prohibited from using your input or output data to train, fine-tune, or improve their models.
  • No data retention beyond processing: Providers must not retain your data beyond the time needed to generate a response (typically seconds).
  • Incident notification: Providers must notify SecureAI of security incidents affecting customer data within defined timeframes.
  • Audit rights: SecureAI retains the right to audit provider compliance with these terms.

Supported Model Providers

Provider API Endpoint Region Data Retention Policy
OpenAI United States No retention (API terms, not consumer terms)
Anthropic United States No retention under commercial API terms
Azure OpenAI Configurable (matches your Azure region) No retention; data processed in your selected Azure region
Local models (Ollama, vLLM) Your infrastructure Data never leaves your network

Note: When using local models hosted on your own infrastructure, no conversation data is sent to any external service. This is the most restrictive data handling option available.

Controlling Model Provider Access

Administrators can restrict which model providers are available to users:

  • Provider allowlisting: Enable only the providers that meet your compliance requirements.
  • Model-level restrictions: Within a provider, restrict which specific models are available (e.g., allow GPT-4 but not GPT-3.5).
  • Default model assignment: Set a default model that meets your security baseline so users do not need to select one manually.

For configuration instructions, see Adding Custom Model Providers.

Network Security

Perimeter Controls

  • Cloud Armor WAF: Protects against OWASP Top 10 threats including SQL injection, XSS, and request smuggling. Custom rules can be configured for your deployment.
  • DDoS protection: GCP's built-in DDoS mitigation at the network and application layers.
  • IP allowlisting: Organizations can restrict access to specific IP ranges or CIDR blocks. See Setting Up IP Allowlisting for configuration.

Internal Network Security

  • VPC isolation: SecureAI services run in a dedicated Virtual Private Cloud with no public IP addresses on application instances.
  • Private service connectivity: Cloud SQL and Cloud Storage are accessed via private IP addresses within the VPC, not over the public internet.
  • Egress restrictions: Outbound traffic is limited to allowlisted destinations (model provider APIs, GCP services). Unauthorized outbound connections are blocked by VPC firewall rules.

Identity and Access Management

Service-Level IAM

SecureAI follows the principle of least privilege for all service accounts:

  • Each Cloud Run service operates with a dedicated GCP service account.
  • Service accounts have only the permissions required for their function (e.g., the API server can read/write Cloud SQL but cannot access Cloud Storage directly).
  • Service account keys are not used. Workload Identity provides keyless authentication between services.

User Authentication

SecureAI supports multiple authentication methods:

Method Use Case
SAML SSO Enterprise organizations with an identity provider (Okta, Azure AD, Auth0)
OIDC SSO Organizations using OpenID Connect-compatible providers
Local accounts Development or small deployments without an identity provider

For SSO configuration, see How to Configure SAML SSO and How to Configure OIDC SSO.

Session Security

  • Sessions are stored server-side. Session tokens are opaque identifiers that reference server-side state.
  • Session tokens are transmitted in HTTP-only, Secure, SameSite cookies.
  • Configurable session timeout (default: 24 hours of inactivity).
  • Administrators can terminate individual user sessions from the admin panel.

Secrets Management

Application secrets (database credentials, API keys for model providers, encryption keys) are stored in GCP Secret Manager:

  • Secrets are encrypted at rest using envelope encryption (AES-256).
  • Access to secrets is controlled by IAM policies — only the specific service account that needs a secret can access it.
  • Secret versions are immutable. Updates create new versions, and old versions can be disabled or destroyed.
  • Secret access is audit-logged — every read is recorded with the identity of the accessor, timestamp, and secret version.

Vulnerability Management

  • Dependency scanning: Container images are scanned for known vulnerabilities before deployment using GCP Artifact Registry vulnerability scanning.
  • Runtime scanning: Cloud Run services are monitored for runtime anomalies.
  • Patch cadence: Critical vulnerabilities (CVSS 9.0+) are patched within 48 hours. High vulnerabilities (CVSS 7.0-8.9) are patched within 7 days.
  • Penetration testing: External penetration tests are conducted annually. Summary reports are available under NDA — contact your account representative.

Incident Response

SecureAI maintains a documented incident response plan:

  1. Detection: Automated alerting on anomalous patterns (failed authentication spikes, unusual data access, WAF rule triggers).
  2. Triage: On-call engineering assesses severity within 15 minutes of alert.
  3. Containment: Affected services are isolated. If a model provider is compromised, traffic to that provider is halted.
  4. Notification: Affected organizations are notified according to contractual SLAs (typically within 24 hours for confirmed data incidents).
  5. Recovery: Services are restored from known-good state. Root cause analysis is completed and shared with affected organizations.
